01 / como ver o site

VER O SITE.

Tem três formas. Da menor pra maior fricção.

A. Direto no GitHub Pages (já está no ar)

O repositório está configurado pra publicar automaticamente em todo push pra main. URL pública:

https://kaluanbernardo.github.io/personal-website/

Pra acompanhar o status do deploy, abre Actions no repo do GitHub e procura a última run "Deploy". Se for primeira execução, garanta em Settings → Pages → Source: GitHub Actions.

B. Local em modo dev (com hot-reload)

No terminal, dentro da pasta do projeto:

# instala dependências (só na primeira vez)
npm install

# sobe o servidor de desenvolvimento
npm run dev

Abre no navegador: http://localhost:4321/

Esse modo recarrega sozinho a cada arquivo salvo. Ideal pra editar conteúdo.

C. Local imitando produção

npm run build
npm run preview

Abre http://localhost:4321/personal-website/. Esse usa exatamente a mesma base de URL do GitHub Pages — útil pra testar links antes de subir.

▶ Atalho diário

Pra editar conteúdo, o fluxo é: roda npm run dev, edita o arquivo, salva, vê no navegador. Quando estiver bom, comita e dá push — o deploy é automático.

02 / domínio próprio

DOMÍNIO PRÓPRIO.

5 passos. Demora ~1h ativa + até 24h de propagação DNS.

1. Comprar o domínio

  • .com.br: registro.br (mais barato, ~R$ 40/ano).
  • .com / .me / .dev: Cloudflare Registrar, Namecheap, Porkbun (preço de custo, sem upsell).

2. Configurar o DNS

No painel da onde você comprou (ou no Cloudflare DNS, se levar pra lá), aponte para o GitHub Pages:

Para o domínio raiz (seudominio.com) — adicione 4 registros A:

A   @   185.199.108.153
A   @   185.199.109.153
A   @   185.199.110.153
A   @   185.199.111.153

Para o www (www.seudominio.com) — um CNAME:

CNAME   www   kaluanbernardo.github.io.

3. Adicionar arquivo CNAME no repo

Crie o arquivo public/CNAME com uma única linha — o seu domínio:

seudominio.com

Isso garante que o GitHub Pages saiba qual domínio responder.

4. Ajustar a config do Astro

Em astro.config.mjs, troque o bloco base pra ficar na raiz:

// antes:
base: isDev ? '/' : '/personal-website/',

// depois:
base: '/',
site: 'https://seudominio.com',

Sem isso, os links continuariam apontando pra /personal-website/....

5. Configurar nas Settings do GitHub

No GitHub: Settings → Pages → Custom domain → digite seudominio.com → salvar.
Aguarde alguns minutos e marque Enforce HTTPS.

★ Recomendação

Se possível, mude o DNS pra Cloudflare (gratuito) — fica mais rápido, ganha proteção contra ataques e os relatórios de tráfego. Você só muda os nameservers no registrador e configura tudo no painel deles.

DNS pode demorar até 24h pra propagar mundialmente. Pra testar, use dig seudominio.com ou dnschecker.org.

04 / cms keystatic

EDITAR VIA CMS.

O modo mais fácil. Não precisa abrir nenhum arquivo de texto.

Acessar o painel

Com o servidor de desenvolvimento rodando (npm run dev), abre:

http://localhost:4321/keystatic

Vai aparecer a tela de login. Selecione "Local" — você está editando os arquivos diretamente.

⚠ Importante

O Keystatic só está disponível em modo dev (não vai pro site público). É uma interface pra editar arquivos locais. Você ainda precisa comitar e dar push pra publicar.

O que dá pra editar pelo painel

  • Posts (do blog) — título, descrição, data, tag, idioma, marcar como rascunho/destaque, conteúdo em rich text.
  • Projetos — título, tipo, ano, veículo/local, URL externa, descrição.
  • Publicações acadêmicas — título, venue, ano, tipo (artigo/capítulo/livro/tese/dissertação), URL.

Fluxo: criar um novo post

  1. No painel, clique em Posts → "+ Add".
  2. Preenche os campos (título, descrição, data, tag, tempo de leitura).
  3. Marca o idioma (PT ou EN).
  4. Escreve o corpo no editor.
  5. Clica em Save.
  6. O Keystatic gera o arquivo .mdoc em src/content/posts/.
  7. Comita e dá push pra publicar:
git add src/content/posts/
git commit -m "novo post: título"
git push

Editar um post existente

No painel, clica no post na lista, faz a edição, salva. Comita e push.

Deletar

No painel, abre o item e clica em Delete. O arquivo é removido. Comita e push.

05 / textos da interface

TEXTOS UI.

O Keystatic gerencia conteúdo. Pros textos do menu, hero, botões, etc., tem um arquivo central.

Tudo está em src/i18n/ui.ts. É um único arquivo com dois objetos: pt e en. Tem mais ou menos 530 linhas, mas é só prosa.

Exemplo: trocar a chamada da home

// src/i18n/ui.ts — bloco "home" do português
home: {
  eyebrow: '— site pessoal · ed. 2026',
  title: { line1: 'Nome', line2: 'Sobrenome.' },
  intro:
    'Product manager em educação, jornalista freelancer e ...',
  ctaBlog: 'Ler o blog →',
  ctaCV: 'Baixar CV',
  ...
}

Trocar o nome em todo lugar

Procura por 'nome sobrenome' (PT) e 'first last' (EN) no arquivo e substitui. Esses entram automaticamente no header, no monograma, na tag NS//26, etc.

Mensagens da newsletter, do blog, da contato

Dentro de ui.ts tem um bloco por seção:

  • home — hero, cards, "do blog"
  • sobre — eyebrow, título, parágrafos, "agora"
  • cv — experiência, skills
  • academico — linhas de pesquisa, formação
  • blog — título, filtros, copy da newsletter
  • projetos — título, subtítulo, filtros
  • contato — texto, e-mail, redes sociais
  • footer — texto do rodapé
▶ Como editar

Abre o arquivo num editor de texto, encontra o bloco da seção, troca a string entre aspas. Salva. Se o npm run dev estiver rodando, recarrega sozinho.

Versão em inglês fica logo abaixo, dentro do mesmo arquivo, no objeto en. Mantém os dois sincronizados.

  • um arquivo: salva em public/favicon.png e usa href="/favicon.png".

10 / publicar

PUBLICAR.

Tudo é git. Push pro main dispara deploy automático.

Fluxo padrão

# 1. ver o que mudou
git status

# 2. adicionar os arquivos
git add src/content/posts/novo-post.mdoc
# ou tudo de uma vez:
git add -A

# 3. comitar com uma mensagem clara
git commit -m "Novo post: economia da atenção"

# 4. push pro GitHub
git push

Em ~1-2 minutos o GitHub Actions compila e publica. Acompanhe em Actions no repo.

Trabalhando em uma branch

Pra mudanças maiores (ex.: trocar paleta de cores, redesenhar uma página), use uma branch:

git checkout -b experimento-cores
# edita o que quiser
git add -A && git commit -m "experimento: paleta verde"
git push -u origin experimento-cores

Abra um Pull Request no GitHub. Quando estiver bom, clica em Merge. O deploy só roda no main.

⚠ Cuidado

Antes de comitar, rode npm run build. Se passar sem erros, está tudo certo. Se quebrar, o deploy também vai quebrar.

Reverter uma mudança

Se publicou algo que não queria:

git revert HEAD       # gera um novo commit que desfaz o último
git push

Não use git reset --hard em commits já publicados — você apaga histórico.

11 / sos

QUANDO QUEBRA.

Os problemas mais comuns e como resolver.

"npm run dev" falha com 'astro: not found'

As dependências não foram instaladas. Roda:

npm install

Página em branco / 404 ao abrir local

Em modo dev, o site roda em http://localhost:4321/ (sem o /personal-website/). Se você abriu http://localhost:4321/personal-website/, vai dar 404. Tira o /personal-website da URL.

Mudei o texto e nada apareceu

  • Confirme que salvou o arquivo.
  • Confirma que npm run dev ainda está rodando.
  • Recarrega a página com Ctrl+Shift+R (force reload).

Build falha no GitHub Actions

Abre Actions no repo, clica na run com X vermelho, lê o erro.

  • Erro de tipo: rode npx astro check local pra ver onde.
  • Erro de schema: provavelmente um campo obrigatório faltando num .mdoc (ex.: readingTime).
  • Erro de import: provavelmente um arquivo deletado mas ainda referenciado.

HTTPS do domínio próprio não habilita

Esperar 1-2h depois de configurar. Se persistir, em Settings → Pages, clica em Remove no domínio, espera 5min, adiciona de novo.

Keystatic mostra "Cannot connect"

Roda npm run dev antes de abrir /keystatic. O CMS é parte do servidor de dev.

Quero mudar o ano "EDIÇÃO 04 · 2026"

Arquivo: src/components/pages/HomePage.astro — procura por EDIÇÃO 04 e edita.

Para tudo o resto

Em qualquer dúvida específica ("como mudo X?"), me pergunte com a captura de tela ou o nome da seção. Posso aplicar a mudança e gerar um diff.
